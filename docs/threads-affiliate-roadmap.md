# Threads アフィリエイト 自動化ロードマップ（10アカウント運用）

Threads（Meta）でアフィリエイトを自動化し、**10アカウントを持続的に並行運用**する計画。
既存のブログ自動化資産（Gemini生成・アフィリリンク生成・候補クロール・cron・設定UI・
媒体プロファイル基盤 `core/media.py`）を流用する。

---

## 大前提（守らないと全滅する）
1. **持続運用＝「別ジャンル・別人格の良質アカウント×10」**。同一内容のクローン乱立は即BAN。
   自動化は「生成・予約投稿・分析」の裏方に徹し、各アカウントは独立したコンテンツを持つ。
2. **ステマ規制**：全アフィリ投稿に `#PR` / `#広告` 等の広告明示を**自動で必ず付与**。
3. **Amazonアソシエイト/もしも規約**：アソシエイト表記、リンク偽装禁止。クリック計測は
   各ASPの管理画面 or 計測リンクで行う（Threadsはクリック数を返さない）。
4. **Meta自動化ポリシー**：人間らしい投稿頻度、急なスケール禁止、アカウント温め(warming)。
   1アカウントずつ実績を作ってから増やす。

---

## フェーズ構成

### Phase 0 — 実現性検証＆コンプラ設計（まず1アカウント）
**目的**: 「Threadsアフィリは自分の商材で成立するか」を最小コストで検証。
- Threadsプロアカウント1個＋Instagram連携＋**Meta開発者アプリ作成**→Threads APIトークン取得。
- 手動〜半自動で**10〜20投稿**し、反応・クリック・発生を観測（フック型／レビュー型／比較型を試す）。
- 投稿フォーマット確定：**フック1行 → 価値（用途/失敗回避）→ CTA＋計測リンク → #PR**。
- 決定: ジャンル候補、1投稿の型、計測方法（もしも計測リンク等）。
**Gate**: 1アカウントで「クリック→発生」が1件でも出るか。出ないなら型/ジャンルを練り直す。

### Phase 1 — コンテンツ自動生成（Threads用）
**目的**: 既存Geminiパイプラインを「短文Threads投稿」用に転用。
- 新プロンプト: 長文記事ではなく**Threads向け短文**（280字目安・フック・絵文字・改行・CTA・#PR）。
  `core/gemini_client.py`（REST・モデル切替UI）をそのまま利用。
- 商品ソース: 既存の**ランキング/楽天ジャンル/参照元クロール**を流用して投稿ネタを供給。
- アフィリリンク: 既存の **Amazon/楽天/もしも** リンク生成を流用（＋計測リンク化）。
- 出力: 「投稿テキスト＋画像（任意）＋リンク＋ジャンル」を投稿キューに積む。
**成果物**: `core/threads_content.py`（投稿案ジェネレータ）、投稿テンプレ集。

### Phase 2 — 投稿自動化（1アカウント・Threads API）
**目的**: 生成→予約→自動投稿の一気通貫を1アカウントで確立。
- `core/threads_client.py`: Threads API で **投稿作成（テキスト/画像/リンク）＋インサイト取得**。
- 予約スケジューラ: cron（Xserver）で**人間らしい時間帯・間隔**に投稿（1日数件、固定間隔回避）。
- ログ: 投稿ID・時刻・リンク・インサイト（views/likes/replies）を記録。
- 承認フロー（任意）: 既存の review UI 同様、**自動投稿 or 人が承認してから投稿**を選べる。
**Gate**: 1アカウントが警告/凍結なく2〜4週間安定して回るか。

### Phase 3 — アカウントプロファイル基盤（多アカウント化）
**目的**: `core/media.py`（媒体プロファイル）を **アカウントプロファイル**へ一般化。
- `Account`: id / ニッチ / ペルソナ（口調・絵文字）/ Threadsトークン / 投稿スケジュール /
  アフィリタグ / 有効フラグ。**トークン等の機密は env**（`THREADS_TOKEN_<id>`）。
- アカウント別: 投稿キュー・ログ・インサイトを**ID名前空間で分離**（混在防止）。
- 長期トークンの更新（Threadsトークンは期限あり→自動リフレッシュ）。
**成果物**: `core/accounts.py`、アカウント設定スキーマ。

### Phase 4 — 10アカウント・オーケストレーション＆安全運用
**目的**: 10アカウントを**BANされず**回す統制レイヤ。
- スケジューラが全アカウントの投稿を**分散・ずらし投稿**（同時刻一斉を避ける）。
- アカウントごとに**独立したニッチ・人格・商品**（重複コンテンツを作らない仕組み＝近似投稿検出）。
- 安全制御: 1日あたり投稿上限、温め期間（新規は少量から）、警告/凍結の検知アラート。
- 段階スケール: 1→3→6→10 と**実績を見ながら増やす**（一気に10個立ち上げない）。
**Gate**: 3アカウントが安定 → 6 → 10 へ。

### Phase 5 — 分析・最適化ダッシュボード
- アカウント別: 投稿数・インプレ・クリック（ASP）・発生・収益。
- 勝ちパターン抽出（型/ジャンル/時間帯/商品）→ 生成プロンプトに反映するループ。
- 既存の AI設定/統計UI に「Threads」タブを追加して一元管理。

## 決めること（次の意思決定ポイント）
1. **ジャンル設計**: 10アカウントのニッチを何にするか（例: ガジェット/美容/ペット/キャンプ/
   ガジェ女子/在宅/料理…）。**全部バラけさせるほどBAN耐性が上がる**。
2. **Threads API利用可否の実機確認**: Metaアプリ作成→1アカウントでトークン取得→投稿テスト。
   （ここが通らないと全体が始まらないので Phase 0 で最優先）
3. **承認 or 全自動**: 各投稿を人が承認するか、完全自動か（初期は承認推奨）。
4. **計測方法**: もしもの計測リンク / 独自短縮 + リダイレクト計測 のどちらにするか。

## システム設計（推奨アーキテクチャ）

**生成（週1・バッチ）と投稿（毎日・分散ドリップ）を分離**する。これで手間・コスト最小、
かつ「作り置き→人間らしく小出し」でBAN耐性も上がる。

```
[週1バッチ / Xserver cron]
  候補商品(既存クロール) → 選定ゲート(既存selection: 価格/ブランド/薬機法NG)
   → アカウント人格で短文生成(Gemini) → QA(既存qa＋#PR必須)
   → 投稿ドラフトを アカウント別キュー(threads:<id>:drafts)へ
        │
[週1レビュー / 既存スワイプUIを流用・スマホ]
   ドラフトを OK/NG/編集 → OKは予約キュー(threads:<id>:queue)へ
        │
[日次ドリップ投稿 / Xserver cron]
   各アカウント、人間らしい時間帯にキューから1〜数件をThreads API投稿
   → インサイト(views/likes)を threads:<id>:log に記録
```

### コスト（実質ゼロ＝頻度はコストで決めない）
- 短文1投稿 ≈ gemini-3.1-flash-lite で **¥0.06**。10アカ×週7=70投稿/週 ≈ **¥20/月**、
  候補3案生成でも ≈ ¥60/月。画像は商品画像流用で生成コスト0。
- → 頻度は **レビュー手間と鮮度**で決める。**週1生成＋週1レビュー＋日次ドリップ**が最小手間。

### 流用マップ
| 機能 | 流用元 | 変更 |
|---|---|---|
| 商品ソース | 既存クロール | そのまま |
| 選定ゲート(OK/NG前段) | `core/selection` | そのまま |
| 人のレビュー | `/select`・`/review` スワイプUI | 対象=投稿ドラフト |
| QA | `core/qa.py` | 短文用＋#PR必須を追加 |
| 文章生成 | `core/gemini_client` | 短文プロンプト新設 |
| リンク | Amazon/楽天/もしも | ＋計測リンク |
| 多アカ設定 | `core/media.py` | アカウントプロファイルへ一般化 |
| キュー/ログ共有 | overrides(WP共有) | `threads:<id>:*` で名前空間化 |
| 実行基盤 | Xserver cron＋Render | そのまま |

### レビュー単位
**投稿ドラフト単位でOK/NG**（商品選定は自動前段、人は最終の投稿文を承認）。
理由: Threadsは投稿文の質・人格・#PRがBAN/成果を左右するため。初期は全件承認→慣れたら一部自動化。

### スケジューラ設計（人間らしさ）
- アカウントごとに投稿時間帯を分散（例: A=朝/昼/夜の固定枠＋±ランダム、Bは別枠）。
- 同時刻一斉を避ける。新規アカは1日1件から温め、実績で増やす。
- 日次cronが「scheduled_at<=now かつ未投稿」を拾って投稿。

## リスクと姿勢（明記）
- 本計画は **公式Threads APIを使い、規約・法令（ステマ規制）を守る前提**。
- **凍結回避のためのフェイク（なりすまし・自演エンゲージ・フィンガープリント偽装）は行わない。**
  生き残りの本質は「各アカウントが本当に役立つ別コンテンツであること」。
- それでもMetaの判断で凍結はあり得る → **1アカウント集中リスクを避け分散**、段階拡大で守る。
