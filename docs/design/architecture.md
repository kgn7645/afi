# アーキテクチャ設計書 — Issue #29

アフィリエイト記事 自動化システムの全体像。「何が・どこで・どの順に動くか」を1枚で把握するための文書。

## 目的とゴール
- 1日約1時間（レビューのみ）で回す
- 月15万円規模の収益（検索流入の蓄積で達成）
- 非属人・ほぼ全自動。担当者はスマホ/Macで**チェックと公開だけ**

## コンポーネント構成

```
                    ┌─────────────────────────────┐
   レビュー担当      │  Googleスプレッドシート       │   ← 商品キュー（取り上げ商品）
   (Mac/ブラウザ) ──▶│  (キュー: brand,category,...) │      担当者がブラウザで追加
                    └──────────────┬──────────────┘
                                   │ ① 読込(Sheets API)
                                   ▼
   ┌───────────────────────────────────────────────────────────┐
   │  生成ツール（Python） … 小型VPS上で cron 実行（無人）       │
   │   batch.py        : 選定→記事生成→リンク生成→WP下書き      │
   │   index_submit.py : 公開記事を検知→IndexNow送信            │
   │   core/*          : 抽出/選定/生成/リンク/WP/オーケスト     │
   └───┬───────────┬───────────┬───────────┬───────────────────┘
       │②Gemini    │③楽天API   │④もしもリンク│⑤WP REST(下書き作成)
       ▼           ▼           ▼           ▼
   [Gemini]   [楽天商品検索]  [自前生成]   ┌──────────────────────┐
   記事本文    商品URL/画像    a_id埋込     │  WordPress (ouchibase) │
                                          │  下書き → 公開 → 公開面 │
                                          └──────────┬───────────┘
                                                     │⑥レビュー(プレビュー)
   レビュー担当 (スマホ/Mac WordPressアプリ) ─────────┘
        └─ チェックリストで確認 → 「公開」 or 「ゴミ箱」
                                                     │⑦公開を検知
   index_submit.py (cron) ───────────────────────────┘ → ⑧IndexNow(Bing等)
```

## 各コンポーネントの責務
| コンポーネント | 役割 | 動作場所 |
|----------------|------|----------|
| Googleスプレッドシート（キュー） | 取り上げ商品の入力（担当者が編集） | クラウド（ブラウザ） |
| batch.py | キュー→選定→記事生成→リンク→WP下書き。重複スキップ | VPS（cron） |
| index_submit.py | 公開済み記事を検知しIndexNow送信 | VPS（cron） |
| core/* | 各工程のロジック（再利用部品） | VPS |
| Gemini API | 記事本文・タイトル・メタ生成（無料枠） | 外部 |
| 楽天商品検索API | 商品URL/画像/価格の取得（無料・IP許可制） | 外部 |
| もしもリンク（自前生成） | 成果ID(a_id)を埋めたリンクHTML生成 | VPS内ロジック |
| WordPress (ouchibase.com) | 下書き保管・公開・公開サイト配信 | レンタル/サーバー |
| WordPressアプリ | レビュー担当が下書き確認→公開 | スマホ/Mac |
| IndexNow | 公開URLの即時インデックス通知 | 外部 |

## データフロー（標準運用）
1. **入力**: レビュー担当がスプレッドシートに商品（キーワード/ブランド+カテゴリ/URL）を追加
2. **生成（無人・毎日）**: VPSのcronが `batch.py` を実行 → キュー読込 → 各商品で
   楽天検索→もしもリンク生成＋Gemini記事生成 → **WP下書き**作成（既出はスキップ）
3. **レビュー（人・スマホ/Mac）**: WordPressアプリで下書きを[チェックリスト](../checklist-review.md)で確認 → 公開 or 却下
4. **インデックス（無人）**: cronの `index_submit.py` が公開を検知 → IndexNow送信
5. **（将来）実績**: 流入/成果をモニタリング(#20)し、シートへ書き戻し(#4)、選定にフィードバック

## 役割分担（概要 / 詳細は[運用設計書](./operations.md)）
- **管理者**: VPS・鍵(.env)管理、監視/障害対応、生成ルール(config.yaml)調整
- **レビュー担当（Mac）**: キューへ商品追加、下書きレビュー→公開。**Python不要・ブラウザとWordアプリのみ**

## 実行環境（概要 / 詳細は[インフラ設計書](./infrastructure.md)）
- 生成ツールは**固定IPの常時起動環境（推奨: 小型VPS）**で動かす
  - 理由: 楽天APIがIP許可制 → 固定IPが必要。無人運用 → 常時起動が必要
- レビュー担当の端末には何もインストール不要（WordPressアプリ＋ブラウザ）

## 主要な設計判断
- **下書き経由で人が必ず確認**: AIの事実誤り（特に「どこの国の企業か」）対策。既定=draft。
- **キューはGoogleスプレッドシート**: 非エンジニアの担当者がブラウザで編集でき、元運用とも親和（#sheet-queue）。
- **収益リンクはもしも経由（a_id）**: 楽天独自アフィリは使わずURLはクリーン化。
- **インデックスはIndexNow主体**: Googleに一般ページのバルク即時インデックスAPIが無いため。
