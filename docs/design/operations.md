# 運用体制・役割定義書 — Issue #31

誰が何をするかを明文化し、属人性を排除する。

## 役割定義
| 役割 | 担当 | 主な責務 | 必要なもの |
|------|------|----------|-----------|
| **管理者** | あなた | VPS・鍵(.env)管理、生成ルール(config.yaml)調整、監視・障害対応、アカウント発行 | VPSへのSSH、各API管理画面、GitHub |
| **レビュー担当** | 別担当（Mac） | キューへ商品追加、下書きのチェック→公開/却下 | Googleスプレッドシート、WordPress「編集者」アカウント、WordPressアプリ |

> レビュー担当は**Python・サーバー操作不要**。ブラウザとスマホアプリだけで完結する。

## 1日の運用フロー
```
（無人）06:00  VPS cron: batch.py 実行
              → キュー(スプレッドシート)から15件を生成 → WP下書きに積む
              → 既出商品は自動スキップ、失敗は通知

（人）任意の時間（約30〜60分）  レビュー担当: WordPressアプリ
              → 下書きを1件ずつプレビュー
              → チェックリストで確認（事実/薬機法/リンク）
              → OK=公開 / 軽微=直して公開 / 事実NG=ゴミ箱
              → 在庫切れ等に気づいたらシートのstatusを更新

（無人）随時  VPS cron: index_submit.py
              → 公開済みを検知 → IndexNow送信

（人）週次など  レビュー担当 or 管理者: シートに新しい商品を補充
管理者: 流入/成果の数字を確認（#20）し、売れ筋カテゴリをシートへ反映
```

## オンボーディング（新しいレビュー担当）
1. WordPressに「編集者」アカウントを発行（管理者）
2. キュー用スプレッドシートを共有（編集権限）
3. スマホにWordPress公式アプリを入れ、`https://ouchibase.com` でログイン
4. [レビュー手順](../runbook-review.md)と[チェックリスト](../checklist-review.md)を共有
5. 最初の数件を一緒に確認して目線合わせ

## エスカレーション（困ったとき）
| 状況 | 対応 |
|------|------|
| 下書きが増えていない | 管理者へ連絡（バッチ失敗/キュー空の可能性。#21の通知で検知） |
| 楽天カードが表示されない | プレビューで確認（編集画面では出ない）。直らなければ管理者へ |
| 企業情報が怪しい記事が多い | 公開せず却下し、管理者へ報告（company_hint/プロンプト改善 #15） |
| WordPressにログインできない | 管理者がアカウント/パスワード再発行 |

## 品質と非属人化の指針
- チェックは[チェックリスト](../checklist-review.md)に従い、判断を個人の感覚に依存させない
- 同種の問題が繰り返すときは管理者にフィードバックし、ルール/プロンプト側で恒久対処
- 手順・判断基準はすべて本リポジトリのdocsに集約し、口頭引き継ぎに頼らない（#22 Runbookで総括）
