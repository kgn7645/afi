# 常駐運用・エラー通知・ログ監視（Issue #21）

無人でバッチを毎日回し、**失敗を担当者に通知**し、ログを肥大化させない運用設定。

## 1. 通知先（Slack または Discord）を用意
どちらか好きな方の **Incoming Webhook URL** を取得：
- **Discord**: サーバー設定 → 連携サービス → ウェブフック → 新規作成 → URLをコピー
- **Slack**: App「Incoming Webhooks」を追加 → チャンネル選択 → URLをコピー

`.env` に設定：
```
NOTIFY_WEBHOOK_URL=（取得したWebhook URL）
NOTIFY_ON_SUCCESS=true   # 成功時も毎回サマリを送る（= 動いている合図/ハートビート）。falseで失敗時のみ
```

## 2. 通知される内容
- **完了サマリ**: `生成 N / 重複スキップ M / 失敗 K`（失敗・選定NG・警告件数の明細つき）
- **異常終了**: 予期せぬ例外でバッチが落ちた場合、エラーとトレースバックを通知し**非ゼロ終了**

> ハートビート運用：`NOTIFY_ON_SUCCESS=true` にしておくと、毎朝サマリが届く＝届かなければ
> 「cronが動いていない」と気づける（最も簡単な死活監視）。

## 3. cron 設定（Xserver）
サーバーパネルの「Cron設定」か `crontab -e` で：
```cron
# 毎朝6時に最大15件を下書き生成。ログは batch.log に追記（自動ローテーションあり）
0 6 * * *  cd /home/<アカウント>/affiliate-automation && .venv/bin/python batch.py --limit 15 >> data/batch.log 2>&1
```
- バッチは `data/batch.log` が約2MBを超えると**末尾1000行に自動圧縮**（ローテーション）。
- キュー未配置の日は静かにスキップ（異常終了しない）。

## 4. 監視の考え方（軽量）
- **正常**: 毎朝サマリ通知が届く
- **異常**: 「失敗 K>0」や「🔥異常終了」の通知 → 内容を見て対応
- **無通知**: cron自体が動いていない疑い → Xserverのcron設定/パスを確認

## 5. よくある通知例と対応
| 通知 | 原因 | 対応 |
|---|---|---|
| `失敗 K`／本文に `429` | Gemini無料枠切れ（#66） | 課金有効化 or 件数を減らす |
| `WordPress投稿に失敗` | WP/REST/権限 | WPアプリパスワード・REST制限を確認 |
| `🔥異常終了` | 想定外例外 | トレースバックを確認・必要なら修正 |
